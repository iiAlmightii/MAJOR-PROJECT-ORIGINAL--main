1A — ByteTrack Parameter Changes player_tracker.py

Parameter Old New Why
lost_track_buffer 30 frames (~1.5s) 120 frames (~6s) Players can be off-camera during timeouts/rotations
minimum_matching_threshold 0.8 0.65 Allows matching through partial occlusion
track_activation_threshold 0.45 0.50 Reduces false-positive detections from audience/scoreboard
Max tracks hard cap None 16 12 players + 2 refs + 2 coaches — hard ceiling

1B — Post-processing Track Merger new: track_merger.py

Runs once after full pipeline completes. Merges ghost tracks into their real parent.

Step 1 — Merge candidates

For every pair of Player UUIDs, check 3 conditions:
✓ Same team
✓ Time ranges don't overlap (one ends before other begins)
✓ Spatial gap ≤ 0.25 court units (foot positions close)

Step 2 — Merge + consolidate

Re-assign all tracking rows, actions, and analytics from ghost UUID → parent UUID. Delete ghost Player record. Stats accumulate correctly on the surviving player.

Step 3 — Prune overflow

If still >16 players after merge, rank by total frames detected. Keep top 16, delete the rest (these are noise detections with very few frames).

Step 4 — Stable display IDs

Assign sequential display numbers: Team A → #1–#6, Team B → #7–#12, ranked by first-seen frame. Stored as display_number on Player model. Shown in overlay instead of UUID.

1C — Re-analyze Trigger

Backend

POST /api/matches/{id}/reanalyze

Clears: player_tracking, ball_tracking, players, actions, analytics, rallies rows for this match. Resets match status → "pending". Fires CV pipeline in background. Progress via existing WebSocket.

Frontend

"Re-analyze" button in match detail header (admin/coach only). Confirmation dialog: "This will delete all existing tracking data and re-run analysis. Continue?" Shows existing WebSocket progress bar during re-run.
