Design — Section 3 of 3: Court Map Accuracy

Auto-calibrate by default + manual corner override if result looks wrong

3A — Improved Auto-Calibration homography_service.py

Problem today

Hough line detection on the first keyframe only. Broadcast overlays (score bugs, logos) and audience rows confuse the line detector, producing a skewed homography matrix.

Fix

Sample 5 keyframes spread across the first 30 seconds instead of just frame 1. Compute homography for each, then pick the one with the lowest reprojection error (most self-consistent line intersections). Robustly rejects frames with overlays/blurs.

3B — Manual Corner Override UI new: CourtCalibrationModal

A "Fix Court Map" button appears in the match detail header (admin/coach only). Clicking it opens a calibration modal that shows the first clean frame of the video. The user clicks 4 court corners in order: top-left → top-right → bottom-right → bottom-left. Each click places a colored dot. Once all 4 are placed, "Apply" sends them to the existing POST /api/matches/{id}/homography endpoint. The map updates immediately on the next tracking poll.

3C — Mini-map Rendering Fix VideoPlayer.jsx

Problem today

Mini-map draws players using raw court_x/y which is unreliable when homography fails. No visual indicator that calibration may be off. Players clamp to [0,1] silently.

Fix

Color-code dots by team (blue/red, matching overlay). Show player display number inside dot. Add a small ⚠ badge on the mini-map if >30% of players have null court coords — signals bad calibration to the user so they know to use "Fix Court Map".
